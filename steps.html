Packaging Steps

The following steps provide illustrative guidance for using FORAP to design, build, and package a computing PjBL project. They show one practical way to move toward a coherent, adoption-ready package, but instructors can adapt the sequence, emphasis, and level of detail to fit their project and course context.

Steps 1-4: Foundation & Design

Start by defining the project idea in a way that makes its instructional use and reuse clear.

  • Decide whether this is a greenfield vs brownfield project ATT1: Greenfield (students build a new system from scratch) or Brownfield (students modify, extend, test, refactor, or maintain an existing system)
  • Identify the computing domain or discipline the project belongs to ATT4
  • For a brownfield project, decide whether the existing system will come from: a prior instructional codebase, an internally prepared codebase, or an open-source project
  • If using an open-source project, look for a codebase that: is realistic enough to support authentic learning PJP1, is stable and buildable, has documentation and setup instructions that can be improved or supplemented GOAL3, has a scope that can be reduced to fit the course GOAL4, has a code structure that students can understand within the available time ATT2 ATT3

This step establishes the project's basic identity and helps determine whether it can later support reuse and adaptation GOAL2.

Write the project learning objectives first, because they will anchor the rest of the package. When useful, express them through a competency specification ATT5 GOAL1.

  • Write the key project design learning objectives GOAL1
  • You can also write them so they can be: compared with the objectives of other FORAP projects, courses, and curricular expectations
  • Optionally use a structured format such as the competency specification of the CC2020 competency model when useful, since it provides a common way to express knowledge, skills, and dispositions ATT5
  • Make sure the objectives describe what students should actually know and be able to do after the project
  • Keep the objectives specific enough to guide: task design, support preparation GOAL3, assessment design GOAL5

After the learning objectives are clear, define the intended project attributes (which might change depending on how the project evolves).

  • Record whether this is a greenfield or brownfield project ATT1
  • Record the scope and duration ATT2
  • Record the technical complexity ATT3
  • Record the domain or discipline ATT4

These attributes help instructors: understand project fit, compare it with other projects, decide whether adaptation is needed GOAL2, judge whether the level of challenge is appropriate GOAL4.

Now design the internal structure of the project itself. This should happen before detailed support materials are built, because support depends on the tasks students will actually perform.

Common ways of structuring computing PjBL projects include:

  • Software life-cycle phases - requirements, design, implementation, testing, deployment or presentation
  • Subprojects - a sequence of smaller connected tasks or mini-projects
  • Sprint iterations - repeated cycles, often used in agile-oriented courses
  • Software change tasks - feature additions, bug fixes, refactoring, maintenance work
  • Modules - project parts that each contribute to a larger final system

When designing the task structure:

  • Prefer tasks that build on one another rather than unrelated tasks
  • Let the project increase in difficulty progressively GOAL4
  • Make sure the sequence supports: authentic work PJP1, active learning PJP2, self-directed learning PJP3, collaboration PJP4, critical thinking PJP5

This step is also where you decide whether the project should be: a shorter module, a half-semester project, or a full-semester project ATT2.

Steps 5-7: Support Package Creation

Now prepare the materials instructors need in order to adopt, implement, adapt, and sustain the project. This step serves both reuse and adaptability GOAL2 and support for teaching and learning GOAL3.

5.1 Prepare the project overview INS1

Summarize the project and explain how an instructor can adopt and adapt it.

  • What the project is for
  • What kinds of courses it fits
  • Whether the project is greenfield or brownfield ATT1
  • How much time it requires ATT2
  • What technical level it assumes ATT3
  • What learning objectives it supports ATT5
  • What can be simplified, removed, extended, or replaced

5.4 Prepare sample solutions INS4

Create solutions or solution fragments that help instructors evaluate and demonstrate project tasks.

  • Example outputs
  • Completed task versions
  • Instructor-only answer materials
  • Comparison artifacts for grading

5.2 Prepare instructor notes INS2

Document what an instructor needs to know during actual implementation.

  • Teaching notes
  • Common student difficulties
  • Points where students usually need help
  • Suggestions for timing and sequencing
  • Instructional decisions and tradeoffs
  • Past deployment experiences
  • Adaptation advice for different contexts
  • Maintenance notes for dependencies, tools, library versions, setup instructions, external services, links, and codebase areas likely to change

5.3 Prepare the reference course INS3

Show example ways the project can be used in practice.

  • Sample schedules
  • Example semester integration
  • Example sequencing of labs and tasks
  • Alternative implementation pathways
  • Examples of lighter and fuller versions of the project

Once the task structure is defined, identify the support students will need to succeed. This step strongly serves support for teaching and learning GOAL3.

6.4 Prepare task scaffolding STU2 GOAL3

Create project-specific scaffolding that students will use directly while doing the work. Choose the level of scaffolding to balance task challenge with the amount of support and direction students need.

  • Task instructions
  • Milestone checklists
  • Starter code
  • Configuration files
  • Diagrams
  • Troubleshooting guidance
  • Solution hints
  • Design templates
  • Artifact templates

6.1 Prepare installation guidelines STU1

Create explicit setup and access guidance for all required tools and environments.

  • Identify what students must install, configure, access, or connect to
  • For greenfield projects, this may include: languages, frameworks, IDEs, databases, version control, deployment tools
  • For brownfield projects ATT1, also identify: how students will obtain the existing codebase, how they will build and run it, what dependencies it needs, what parts of the existing configuration must be preserved, what local setup problems are likely
  • For complex environments, consider: preconfigured setups, containers, prepared starter repositories, guided setup walkthroughs

6.2 Prepare laboratory exercises STU3 GOAL3

Design labs that help students gain the prerequisite knowledge needed for the project.

  • Identify the prerequisite knowledge students need before or during project work
  • Build labs for: setup and installation practice, learning GitHub or version control, using a framework or API, database access, testing tools, brownfield navigation of an existing codebase
  • Use labs to reduce technical barriers before students reach the larger project tasks GOAL4

6.3 Prepare reference materials STU4 GOAL3

Assemble materials students can consult when they need background or procedural support.

  • Include lecture materials or slide decks for key prerequisite knowledge, then add external references for required tools and installation, APIs, libraries, frameworks, protocols, formats, report templates, and example artifacts
  • Include only the references that are likely to be useful for this project, not a general reading dump

Design assessment so it remains clearly connected to the learning objectives ATT5 and supports both formative and summative uses GOAL5.

7.1 Create phased milestones ASM1

Use milestones to structure student progress and support feedback during the project.

  • Define meaningful intermediate deliverables
  • Make milestone boundaries match the task structure from Step 4
  • Use milestones to: check progress, identify where students are stuck, provide timely feedback, reduce last-minute project collapse

7.2 Create transfer-oriented design problems ASM2

Prepare design problems that assess learning beyond the original project tasks.

  • Use scenario-based prompts
  • Ask students to transfer knowledge to a new context
  • Use them to assess higher-order thinking, especially: analysis, evaluation, synthesis
  • These can be used in: exams, presentations, project reviews, oral discussions

7.3 Create the detailed rubric ASM3

Prepare a rubric aligned with the learning objectives ATT5 ASM3.

  • Define clear evaluation criteria
  • Include criteria for relevant artifacts such as: code, design documents, reports, presentations, demonstrations
  • Make the rubric usable for both feedback and grading GOAL5

Steps 8-10: Review & Refine

Before final packaging, review whether all parts are working together coherently.

  • Check whether the learning objectives GOAL1ATT5 match: the task structure, the project attributes ATT1 ATT2 ATT3 ATT4, the instructor support INS1 INS2 INS3 INS4, the student support STU1 STU2 STU3 STU4, the assessment package ASM1 ASM2 ASM3
  • Check whether the level of challenge remains appropriate GOAL4
  • Check whether sufficient support has been added for both instructors and students GOAL3

This is the point where the project becomes a coordinated FORAP package rather than just a collection of materials.

Organize all materials into a coherent, adoption-ready format.

  • Make the structure easy to navigate
  • Separate instructor-facing and student-facing materials
  • Make adaptation points visible
  • When included, make comparison with other FORAP projects easier through the competency specification ATT5 and project attributes ATT1 ATT2 ATT3 ATT4
  • Preserve reusability and adaptability GOAL2

After classroom use, improve the project package based on evidence from use.

  • Revise supports GOAL3
  • Revise challenge level GOAL4
  • Revise assessment GOAL5
  • Revise instructor notes, including maintenance guidance INS2
  • Update brownfield or open-source setup materials ATT1
  • Refine objectives and attributes when needed ATT1 ATT2 ATT3 ATT4 ATT5
